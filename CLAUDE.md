# Axioms

여러 프로젝트, 여러 기기, 여러 세션에 걸쳐 적용되는 보편 원칙.
실제 시행착오에서 추출했으며, 무시하면 시간과 비용을 낭비한다.

---

## 디버깅

### 동일 시스템인데 결과가 다를 때 — 반드시 이 순서로
1. **데이터 파일 먼저** (vocab, config, tokenizer, 가중치, 매핑 테이블) — 줄 수, 첫 줄, 샘플 인덱스 대조
2. 코드는 그 다음
3. 전처리/연산 로직은 마지막

코드에 버그가 있을 확률보다 데이터 파일이 어긋나 있을 확률이 훨씬 높다. 코드는 컴파일러가 검증하지만 데이터 파일은 아무도 검증하지 않는다.

### 파일 생성/변환 직후 — 즉시 검증
원본 데이터에서 파생 파일(vocab.txt, config, 룩업 테이블 등)을 생성했으면:
1. 총 항목 수가 원본과 일치하는지
2. index 0 (첫 번째 항목)이 올바른지
3. 2~3개 샘플 인덱스를 원본과 교차 확인

생성 직후 10초 검증이, 나중에 이틀 디버깅을 방지한다.

### 가장 단순한 가설부터
복잡한 원인(라이브러리 내부 동작, 하드웨어 차이, 부동소수점 정밀도)을 의심하기 전에 단순한 것(파일 누락, 오타, 인덱스 오프셋)을 먼저 확인한다.

## 작업 방식

### 효율
- 한 문장으로 될 걸 세 문장으로 쓰지 않는다.
- 방금 한 작업을 요약하지 않는다 — diff를 보면 안다.
- 유저가 화났을 때 웃거나 가볍게 넘기지 않는다.

### 변경 최소화
- 요청받은 것만 수정한다. "ついでに" 리팩토링하지 않는다.
- 동작하는 코드에 독스트링, 타입 힌트, 에러 핸들링을 임의로 추가하지 않는다.
- 한 줄로 고칠 수 있으면 한 줄만 고친다.

## 이 문서의 운영 규칙

- 여기에는 **모든 프로젝트에 통용되는 원칙만** 넣는다. 특정 프로젝트(모델명, 파일 경로, API 이름 등)에 종속되는 내용은 해당 프로젝트의 CLAUDE.md에 넣는다.
- 새로운 원칙은 실제 실패 경험에서만 추출한다. 이론적으로 좋아 보이는 것은 넣지 않는다.
- 기존 원칙이 틀렸거나 불필요하면 삭제한다. 늘리기만 하지 않는다.
